i.Time Series DBs can compress data by a large percentage as they essentially use the
timestamp or the time column to their advantage and make use of compression algorithms to effectively use less data to represent the same data that was 
available before. For example, if we have a heartbeat column, timescale will essentially just store the first heartbeat actual value and the deltas to it
 thereafter for the latter timestamps essentially saving up a few bits/bytes of data due to bitpacking. For a large dataset, this adds up. For other datatypes 
 like strings it can use other techniques like dictionary plus run length encoding etc. Basically, TimeSeries databases like TimeScale can compress datapoints
  well which are repeated( through something like run length encoding), change gradually and predictably(delta, delta of delta, gorilla encoding). TSDBs also 
  use columnar storage where instead of storing row values they store columnar values which makes it easier for the compression algorithms to work.

Health Data like heart rate, spo2, steps, and distance are great for compression as for a 1 second resolution they change minimally or not at all which makes
 them repetable and high compressable. TimeScale can then make effective usage of something like delta encoding or run length to compress the data significantly. 
 On the other hand, if we have datapoints that produce deltas which are highly random then the compression would not be ideal. For example if we have a datapoint
  going from 0 to 100, storing the delta of 100 would do us no good as it is the same as storing the value itself. Both would essentially take the same amount to store.
   Ideally, deltas which are small, repeating and patterned are best for compression.

5a . To maintain high ingestion speeds, and query performance for multiple users across
different time ranges, a feasible limit might be 128 GB of RAM, with 300TB NVme storage with a 64
core CPU.

5b. For horizontal scaling, we would probably need to divide our data into shards which would live
on multiple servers. This would mean that we would need a strategy -- how do we decide where the
incoming ingestion data goes(on which server)? Furthermore, while querying we would also need to know
where a certain piece of data is so that we can successfully access it. We would need to look
at partitioning strategies and ways to retrieve the needed data(hash, partition key etc.).
We'd also need to decide if we shard the data by time or a particular column(like metrics).
Finally, if a query needs to access multiple shards, we might have to dedicate one node altogether to
make sense of what comes out of all the shards to build the final result of the query.
